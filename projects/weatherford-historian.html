Weatherford Historian

Lead Platform Architect & DevOps Engineer · Weatherford (via Umbrage)
SHEET: A-02
PLATFORM: AWS EKS
DATE: Jul–Sep 2025

Migrated Weatherford's Historian application off hand-applied Kubernetes manifests and onto a standardized, GitOps-managed platform — and left the Weatherford team able to run it themselves.

The Challenge

8+ services, no shared pattern

Historian's microservices were each deployed with their own hand-applied manifests, no two quite alike. Deployments were manual and error-prone, observability was thin, and the Weatherford team had a real gap in Kubernetes and GitOps experience — anything built here had to be simple enough for them to inherit.

Solution Architecture

Standardize, then hand it to Git

Converted every service to a standardized Helm chart, then put FluxCD in charge of deployment so Git became the single source of truth — drift gets detected and reconciled automatically instead of chased down manually. Layered Prometheus, Grafana, and AWS Container Insights on top for observability across dev, QA, staging, and production. Ran a four-week hands-on training program so the Weatherford team could operate the platform without escalating every change back to Umbrage.

GitOps workflow (simplified)

git commit→ FluxCD→ detect drift→ helm upgrade→ EKS cluster
legacy kubectl manifests⇢ standardized to ⇢ Helm + Flux

Key Accomplishments

Outcomes

  • 8+ microservices converted to standardized Helm charts
  • Deployment time cut from 2–4 hours to 30–45 minutes
  • Deployment failure rate down from 20–30% to under 5%
  • Weatherford team now handles 60–70% of requests independently

Lessons Learned

What held up

Standardizing on Helm first made every service after the first one faster to onboard. Migrating stateful workloads needed custom handling that no template covers. FluxCD's learning curve was real — the training program is what actually made GitOps stick with the team, not the documentation alone.
